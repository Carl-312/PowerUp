## 1. 产品概述

**产品名称：** PowerUp

**一句话描述：** 给你的 AI Agent 吃个蘑菇，帮助用户发现、筛选并理解可接入的 Agent 能力扩展。

**V1 交付形态：** 编辑精选、只读浏览的能力目录站点。

**核心价值：**

- 对非技术用户：快速看懂一个能力扩展是什么、适合什么场景。
- 对 AI Builder：快速判断一个能力扩展值不值得接入、怎么接入、去哪看原始资料。
- 对 PowerUp 团队：建立高质量目录基线，为后续开放共建打地基。

---

## 2. 目标用户与核心场景

| 用户角色 | 当前痛点 | PowerUp 提供的价值 |
| --- | --- | --- |
| AI 新手 / 产品经理 | 不知道有哪些 Skill / MCP，可用性判断成本高 | 用通俗信息先帮助理解，再逐步暴露技术细节 |
| AI Builder / 独立开发者 | 信息分散在 GitHub、官方文档、社区帖子中 | 聚合基础信息、接入方式、支持平台和外部链接 |
| 内容编辑 / 站点维护者 | 需要统一口径收录和维护目录内容 | 提供一致的收录标准、字段规范和核验要求 |

### 核心 JTBD

- 当我想给 Agent 增加能力时，我想快速找到可靠候选项，这样我能缩短调研时间。
- 当我看到一个 Skill / MCP 时，我想知道它解决什么问题、适用于哪些平台、如何接入，这样我能决定是否继续深入。
- 当我维护目录内容时，我想用统一标准录入和更新条目，这样站点内容质量稳定可控。

## 3. V1 目标与非目标

### V1 目标

- 建立高质量的 Skill / MCP 精选目录，成为用户发现 Agent 能力扩展的第一入口。
- 提供清晰的分类、筛选、排序和搜索体验，降低发现成本。
- 为每个条目提供结构化详情页，帮助用户快速判断适用性与接入成本。
- 建立最小可用的内容治理规范，保证上线内容可信、可维护、可持续更新。

### V1 非目标

- 不做用户体系，包括登录、注册、OAuth。
- 不做评论、评分、收藏、排行榜等依赖账户的互动功能。
- 不做用户提交内容和审核后台。
- 不做在线试用、Playground、沙盒执行。
- 不做复杂个性化推荐。
- 不把暗色模式、Storybook、CMS 后台作为 V1 必做项。

## 5. 功能需求

### 5.1 首页 / 列表页

首页是 PowerUp 的主入口，承担发现、比较、初筛三件事。

**必须提供：**

- Hero 区，包含站点标题、一句话说明、主搜索框。
- Skill 卡片列表。
- 类型筛选、分类筛选、排序切换。
- 分页导航。
- 空状态。

**卡片必须展示：**

- 名称
- 图标或默认占位图标
- 一句话简介
- 类型标签
- 一级分类标签
- 作者 / 维护组织
- 最后更新时间

**当前实现说明：**

- 当前代码保留了 `icon` 字段，但首页卡片尚未单独渲染图标 UI；判断当前实现时以 `../docs/**` 与真实代码为准。

**交互规则：**

- 默认展示 `published` 状态条目。
- 默认按 `updated_at` 倒序排序。
- 首页每页展示 24 条。
- 筛选、排序、分页都通过 URL 参数驱动，支持刷新与分享链接。
- 点击卡片任意主要区域进入详情页。

**当前实现说明：**

- 当前列表页已提供标题链接和“详情”按钮进入详情页，但未实现整卡任意主要区域都可点击的命中区。

**验收标准：**

- `AC-HOME-01` 首页首屏可直接浏览已发布条目，无需登录或前置操作。
- `AC-HOME-02` 每张卡片完整展示规定字段，不暴露内部治理字段。
- `AC-HOME-03` 首页支持 `type`、`category`、`sort`、`page`、`q` 五类 URL 参数。
- `AC-HOME-04` 首页分页稳定可用，默认每页 24 条。
- `AC-HOME-05` 当筛选或搜索无结果时，显示明确空状态和清空条件入口。

### 5.2 搜索

搜索是首页和全站导航中的共用能力。

**搜索范围：**

- `name`
- `summary`
- `description`
- `tags`

**搜索行为：**

- 顶部导航和首页 Hero 共用同一套搜索行为。
- 首版使用普通 GET 搜索提交，不做即时建议面板。
- 全站任意搜索提交都进入 `/?q=关键词` 结果态。
- 用户按回车后跳转到 `/?q=关键词`，完整结果复用首页列表态，不新增独立 `/search` 页面。
- 搜索结果基于基础字段匹配实现，不把 FTS5、高亮片段或实时建议作为当前 V1 前置条件。

**空状态与异常：**

- 无结果时明确提示“没有找到匹配项”。
- 搜索词为空时回到默认列表态，不发起专用搜索请求。
- 搜索结果不得返回未发布条目。

**验收标准：**

- `AC-SEARCH-01` 顶部搜索框和首页搜索框行为一致。
- `AC-SEARCH-02` 基础搜索范围覆盖名称、简介、标签、描述内容。
- `AC-SEARCH-03` 回车或提交后进入 `/?q=` 结果态，而不是独立搜索页。
- `AC-SEARCH-04` 搜索结果只包含 `published` 状态条目。
- `AC-SEARCH-05` 搜索无结果时显示明确空状态和清空条件入口。

### 5.3 Skill 详情页

详情页用于帮助用户完成“是否值得继续接入”的判断。

**页面区块：**

- 基础信息区
- 详细描述区
- 集成信息区
- 外部链接区

**基础信息区必须展示：**

- 名称
- 图标
- 类型标签
- 一句话简介
- 作者 / 维护组织
- 最后更新时间
- 开源协议（如有）
- 支持平台（如有）

**当前实现说明：**

- 当前详情页头部尚未单独渲染图标 UI；当前已落地字段展示以 `../docs/**` 与真实代码为准。

**详细描述区必须支持：**

- Markdown 渲染
- 适用场景说明
- 标签展示

**集成信息区规则：**

- 有安装说明时展示安装 / 接入方式。
- 有配置示例时展示代码块。
- 仅 `mcp_server` 类型在有数据时展示 API 端点相关信息。

**当前实现说明：**

- 当前 schema 与查询层没有独立的 endpoint 字段，因此现阶段详情页的“集成信息区”实际只围绕 `install_guide`、`config_example` 与公开外链展开。

**外部链接区规则：**

- GitHub、官方文档、相关资源按有值显示，无值不占位。

**验收标准：**

- `AC-DETAIL-01` 详情页按四个信息区组织内容，结构稳定清晰。
- `AC-DETAIL-02` 基础信息区完整展示前台公开字段。
- `AC-DETAIL-03` Markdown 描述和代码示例可正常渲染。
- `AC-DETAIL-04` 外部链接区仅显示有值链接，不展示空占位。
- `AC-DETAIL-05` 访问不存在或未发布 slug 时返回 404。

### 5.4 分类页

分类页是列表页的窄化视图，用于用户沿领域继续探索。

**页面规则：**

- 路由使用分类 slug。
- 页面标题展示 emoji、分类名称、条目数量。
- 复用首页的卡片、排序、分页样式。
- 分类页内不允许切换到其他分类，只允许切换 `type`、`sort`、`page`。

**验收标准：**

- `AC-CAT-01` 分类页路由和分类 slug 一一对应。
- `AC-CAT-02` 分类页展示该分类的名称、emoji 和数量。
- `AC-CAT-03` 分类页复用首页列表交互，但固定当前分类上下文。
- `AC-CAT-04` 非法分类 slug 返回 404。

### 5.5 关于页

关于页用于解释 PowerUp 是什么、为什么存在、如何联系维护者。

**必须包含：**

- 项目介绍
- V1 目标与边界摘要
- 联系方式
- 贡献说明占位文案

**验收标准：**

- `AC-ABOUT-01` 关于页为静态内容页，可被搜索引擎抓取。
- `AC-ABOUT-02` 文案清楚说明当前 V1 是编辑精选目录，不是开放提交平台。

## 6. 收录范围与内容治理

### 6.1 收录类型

| 类型 | 收录 |
| --- | --- |
| MCP | 是 |
| Anthropic 风格 Skill | 是 |
| GPTs / Agent 模板 | 否 |
| Prompt 模板 | 否 |
| OpenAI Plugin 风格 API 插件目录 | 否 |

### 6.2 收录准入标准

一个条目要进入 V1 发布态，必须同时满足以下条件：

- 有可访问的权威来源，至少一个 `source_url`。
- 有清晰用途，不能只是模糊概念或营销页。
- 与 Agent 能力扩展直接相关，而不是普通 SaaS 产品介绍。
- 有基本可验证信息，包括作者、类型、分类、功能描述。
- 维护状态可接受，优先收录近 24 个月内仍有活动或官方维护的项目。

### 6.3 编辑工作流

V1 的内容工作流固定为以下五步：

1. 收集候选项。
2. 校验来源与可访问性。
3. 统一字段口径并写入条目。
4. 复核后置为 `published`。
5. 周期性复核并更新 `last_verified_at`。

### 6.4 字段边界

**前台公开字段：**

- `name`
- `slug`
- `type`
- `icon`
- `summary`
- `description`
- `category`
- `tags`
- `author`
- `license`
- `github_url`
- `doc_url`
- `install_guide`
- `config_example`
- `supported_platforms`
- `updated_at`

**内部治理字段：**

- `source_url`
- `source_kind`
- `last_verified_at`
- `review_status`
- `created_at`

**状态约定：**

| review_status | 说明 |
| --- | --- |
| `draft` | 已收集，未完成核验 |
| `reviewed` | 已完成核验，待发布 |
| `published` | 前台可见 |
| `archived` | 已下线，不在前台显示 |

**验收标准：**

- `AC-CONTENT-01` 已发布条目必须至少有一个可访问的权威来源。
- `AC-CONTENT-02` 已发布条目必须填写 `source_url`、`source_kind`、`last_verified_at`、`review_status`。
- `AC-CONTENT-03` 只有 `published` 状态条目会出现在前台列表、详情和搜索中。
- `AC-CONTENT-04` 每个条目必须至少覆盖名称、简介、描述、分类、作者五项核心信息。

## 8. 非功能要求

### 8.1 SEO

- 首页、分类页、详情页、关于页都必须有可读 title 和 description。
- 当前 V1 以基础 metadata 为准，不把 `sitemap`、`robots`、结构化数据作为首版阻塞项。

### 8.2 性能

- 首屏不依赖庞大客户端包才能完成核心浏览。
- 当前 V1 不把精确 LCP 预算作为首版阻塞验收项；性能专项调优视真实流量和发布需求再补。

### 8.3 响应式

- 桌面端和移动端都必须可用。
- 首页、详情页、分类页在窄屏下不得出现信息不可达或横向溢出。

### 8.4 基础可访问性

- 键盘可操作搜索框、筛选器、分页与主要链接。
- 表单控件和主要交互元素需要有明确焦点状态。
- 颜色对比和标签文案需要基本可辨识。

**验收标准：**

- `AC-NFR-01` 首页、分类页、详情页在桌面与移动端都可正常浏览。
- `AC-NFR-02` 首页和详情页不依赖庞大客户端包才能完成核心浏览。
- `AC-NFR-03` 所有公开页面具备基础 SEO 元信息。
- `AC-NFR-04` 搜索、筛选、分页与主导航具备基础键盘可用性。

## 9. 成功指标

- 首版开发与验收基线为跑通真实浏览闭环；当前内容基线可先以 18 条样本、覆盖 6 个以上一级分类推进。
- 上线后继续扩充到 50 个高质量条目。
- 100% 已发布条目具备可访问来源和最近核验时间。
- 核心关键词如 `MCP server directory`、`agent skills`、`PowerUp` 的搜索引擎收录属于后续优化目标，不作为当前首版阻塞项。
